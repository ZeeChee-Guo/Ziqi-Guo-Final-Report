\chapter{Experiments}


\section{Experiment Setup}
The experiment was conducted on a local Windows laptop. The machine uses an Intel(R) Core(TM) i7-10510U CPU @ 1.80GHz with 8 logical processors and 16 GB of physical memory. The operating system is Windows 11, 64-bit. The system uses Python 3.9.0 as the runtime environment, and the main Python dependencies include Flask, NumPy, pandas, SciPy, scikit-learn, STUMPY, and TSB-UAD.


\section{Datasets}

Three time-series datasets were used to evaluate the system: ECG, Elec, and Weather. These datasets include anomalies and concept drifts. The ECG dataset is a periodic electrocardiogram dataset from the TSB-UAD benchmark suite. It contains approximately 230K time points and labeled anomalies related to premature ventricular contractions \citep{park2025andri,paparrizos2022tsbuad}. In the AnDri demonstration, abrupt and recurrent drift are injected by transitioning between selected ECG patterns. This setting is useful for evaluating whether a detector can avoid treating a change in normal ECG pattern as an anomaly. The Elec dataset describes electricity usage in New South Wales, Australia. It contains approximately 45K time points collected over 2.5 years, with concept drift caused by weekday-to-weekend changes and seasonal usage variation \citep{park2025andri,bifet2010moa}. Anomalies are injected into the series by modifying selected values. This dataset represents energy-consumption data where normal behavior can change because of calendar and seasonal effects. The Weather dataset contains hourly, geographically aggregated temperature records of European countries from 2017 to 2020. It contains approximately 26K time points and includes naturally occurring gradual concept drift caused by seasonal changes \citep{park2025andri,gelaro2017merra2}. The Weather dataset provides a more natural example of gradual drift in real-world time-series data. Among these three data sets, the proportion of anomalies in the ECG data set is 5\%, while the proportions of anomalies in the other two data sets, namely Elec and Weather, are both 10\%.

\section{Evaluation Metrics}

% \fei{What does this mean technically?  Avoid general statements that are vague.  PResent what results????}


The evaluation compares the final predicted anomaly labels with the ground-truth labels in the ECG dataset. When the ARFF file is loaded, the implementation reads the ECG signal value and its class label. The class label is stored as GroundTruth: \textbf{GroundTruth[i] = 1} means that point \(i\) is a true anomaly, and \textbf{GroundTruth[i] = 0} means that it is normal.

After the adaptive thresholding step, each method has a final threshold. The anomaly score of each point is compared with this threshold to generate a binary prediction. For NormA, SAND, and DAMP, one global threshold is used:
\[\hat{y}_i = 1 \quad \text{if} \quad s_i > \theta.\]
For AnDri, the threshold is associated with the normal pattern assigned to the point:
\[ \hat{y}_i = 1 \quad \text{if} \quad s_i > \theta_{m_i}.\]
Otherwise, \(\hat{y}_i = 0\). The predicted labels \(\hat{y}_i\) are then compared with the ground-truth labels \(y_i = \textbf{GroundTruth[i]}\) to calculate precision, recall, and F1-score. User feedback labels are only used to adjust the threshold; they are not used as the final evaluation labels.

The prediction results are evaluated using precision, recall, and F1-score. Let TP denote true positives, FP denote false positives, and FN denote false negatives. TP is the number of anomaly-labeled points correctly detected as anomalies. FP is the number of normal points incorrectly detected as anomalies. FN is the number of anomaly-labeled points missed by the method.

\[Precision = \frac{TP}{TP + FP}\]

\[Recall = \frac{TP}{TP + FN}\]

\[F1 = \frac{2 \times Precision \times Recall}{Precision + Recall}\]

Precision measures how many detected anomalies are correct. Recall measures how many true anomalies are successfully detected. F1-score combines precision and recall into one value. The experiments also report AUC-ROC. A higher AUC indicates better overall separation between anomalous and normal regions.


\section{User-Feedback Threshold Refinement Results and Analysis}

This section evaluates the effect of the user-feedback-driven threshold refinement process under an oracle feedback setting. In the initial setting, each method first produces point-level anomaly scores, and the initial threshold is obtained by selecting the top-$\alpha$ highest-scoring points as anomaly candidates. In other words, $\alpha$ controls the initial expected anomaly proportion. In this experiment, due to the inability to truly simulate user behavior, ground-truth labels are used to simulate ideal user feedback. We assume that the user has accurately identified all the outliers in the training set. Since a large number of correct anomaly labels are provided to the refinement process, the feedback-refined results are expected to improve over the purely automatic initial threshold. Therefore, the results should be interpreted as an upper-bound style evaluation of the feedback process rather than as a fully automatic detection result. The purpose of this experiment is to examine how much the final anomaly labels can improve when reliable user feedback is incorporated into the thresholding stage.

The improvement is particularly clear on the Elec dataset, where AnDri improves from 0.317 to 0.599, NormA improves from 0.255 to 0.560, SAND improves from 0.248 to 0.583, and DAMP improves from 0.099 to 0.449. Similar improvements are also observed on Weather, where the feedback-refined results improve all four methods.

The results suggest that user feedback is useful as a post-processing refinement mechanism for score-based anomaly detection methods. Each evaluated method first produces anomaly scores, and the thresholding stage converts these scores into binary anomaly labels. When user-confirmed anomaly labels are incorporated, the final binary labels become more reliable. This is especially useful for points whose anomaly scores are not high enough to pass the original threshold but are nevertheless confirmed as anomalies by the user.

\subsection{Discussion and Limitations}

The improvement shown in Table~\ref{tab:t0} should be interpreted with caution. This experiment assumes an ideal feedback environment, where user feedback is simulated using real labels. Because the simulated user provides a number of correct anomaly labels, the feedback-refined results are naturally expected to be better than the initial threshold results. Therefore, this experiment should not be read as evidence that the same level of improvement will always occur in real deployment.

In actual interactive systems, users may not check every relevant point, and their feedback may be incomplete or occasionally inconsistent. The purpose of this experiment is instead to demonstrate the potential value of the feedback workflow under a best-case assumption. The initial threshold represents a fully automatic score-to-label conversion, while the feedback-refined results represent a semi-interactive setting in which user-confirmed labels are retained during the final prediction. Under this assumption, the system can recover anomalous points that cannot be detected by the automatic threshold alone.

Another important observation is that the feedback mechanism is not only beneficial to AnDri, but also helpful for the benchmark methods. This indicates that the refinement process is not dependent on a specific anomaly detector. As long as a method can generate anomaly scores, user feedback can be used to improve the final labeling step.

Overall, this experiment supports the effectiveness of incorporating user interaction into AnDri. The automatic threshold provides the initial prediction, but the user feedback stage enables the system to integrate human confirmation into the final anomaly labels. Although the results represent an idealized feedback scenario, they show that combining user-confirmed labels with threshold refinement can improve the quality of anomaly detection results.

However, the baseline methods produce scores close to or above their thresholds, which can cause them to classify the drift region as anomalous.

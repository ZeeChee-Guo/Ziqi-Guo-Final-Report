This chapter describes the methodological workflow used in this project to detect anomalies in time series with concept drift. The chapter builds on the prior AnDri method, which provides the dynamic normal-pattern model and the anomaly scoring mechanism \citep{park2025andri,park2025adaptive}. The Adjacent Hierarchical Clustering component is treated as part of the prior AnDri method and is not claimed as an individual contribution of this report. The focus here is the end-to-end detection methodology implemented in the demo system: score generation, normal-pattern-aware thresholding, active-learning-based label refinement, and final anomaly decision making.

\section{Problem Definition}

Let \(T=[x_1,x_2,\ldots,x_n]\) be a univariate time series. A subsequence of length \(\ell\) starting at index \(i\) is denoted by \(T_{i,\ell}=[x_i,x_{i+1},\ldots,x_{i+\ell-1}]\). The goal is to assign each point, or each subsequence represented at a point, a binary anomaly label \(\hat{y}_i \in \{0,1\}\), where \(\hat{y}_i=1\) indicates an anomaly and \(\hat{y}_i=0\) indicates normal behavior.

The difficulty is that normal behavior is not stationary. A region that differs from earlier data may be a true anomaly, but it may also be a new or recurring normal pattern caused by concept drift. Therefore, the detection problem cannot be solved only by comparing every point against one fixed global normal model. The method must decide both whether a point has a high anomaly score and whether that score should be interpreted relative to the correct normal context.

\section{Method Overview}

The implemented workflow has four main stages. First, the system converts the input time series into anomaly scores. For AnDri, this stage also returns the normal pattern associated with each point. Second, the user selects one or more training regions and marks a small number of anomalies. Third, the system estimates anomaly thresholds from the score distribution of the user-labeled points. Fourth, when the score distribution is ambiguous, the system requests additional user feedback for candidate points and updates the threshold iteratively.

\section{Anomaly Score Generation}

Each detection method produces a score sequence
\[S=[s_1,s_2,\ldots,s_n],\]
where a larger score indicates stronger evidence of abnormality. In the implementation, scores are normalized to the interval \([0,1]\).

For AnDri, the backend uses the prior AnDri scoring module to learn normal patterns and compute point-level anomaly scores. Each point is also assigned to a normal pattern index:
\[g(i) \in \{1,2,\ldots,m\},\]
where \(g(i)\) identifies the normal pattern that best explains point \(i\). This mapping is important because different normal patterns may have different score distributions. A score that is high for one normal pattern may be ordinary for another. Therefore, AnDri does not use only one global threshold; it estimates thresholds separately for the normal-pattern groups observed in the selected training regions.

For comparison, NormA, SAND, and DAMP are also integrated into the same workflow \citep{boniol2021norma,boniol2021sand,lu2022damp}. These methods generate normalized anomaly scores, but they do not provide the same dynamic normal-pattern grouping used by AnDri. As a result, the baseline workflow applies one adaptive threshold to the selected training region, while AnDri applies normal-pattern-specific thresholds.

\section{Adaptive Thresholding with User Labels}

The system maintains a label flag \(f_i\) for each point. Initially, most points are unlabeled. A user can mark points as anomalous or normal:
\[
f_i =
\begin{cases}
0, & \text{unlabeled},\\
1, & \text{user-marked anomaly},\\
2, & \text{user-confirmed normal}.
\end{cases}
\]
The purpose of the adaptive thresholding step is to estimate a threshold \(\theta\) from the user-marked anomaly scores. For a global detector, the final decision is
\[
\hat{y}_i =
\begin{cases}
1, & s_i > \theta,\\
0, & s_i \leq \theta.
\end{cases}
\]

The demo also allows the user to provide an expected anomaly proportion \(\rho\). This value is used to compute a score cutoff \(q_{\rho}\), corresponding to the boundary of the top \(\rho\) percent of scores in the selected training region. This prevents the threshold from becoming too low when only a small number of anomaly labels are provided.

To model the marked anomaly scores, the backend fits a two-component Bayesian Gaussian mixture model:
\[p(s)=\sum_{k=1}^{2}\pi_k \mathcal{N}(s;\mu_k,\sigma_k^2).\]
Only mixture components with weight greater than a small threshold are treated as active. If fewer than two active components remain, the labeled anomaly scores are considered stable, and the threshold is set using both the mixture statistics and the anomaly-proportion cutoff:
\[\theta = \max(q_{\rho}, \mu-\sigma).\]
Here \(\mu\) and \(\sigma\) describe the active anomaly-score component. This rule keeps the threshold high enough to respect the expected anomaly proportion while still adapting to the distribution of user-confirmed anomalies.



\section{Active Learning for Ambiguous Scores}

When two active Gaussian components are found, the labeled scores are ambiguous. This means the current user labels do not yet separate normal and abnormal score regions clearly. The system then computes the intersection between adjacent Gaussian components:
\[\pi_a \mathcal{N}(x;\mu_a,\sigma_a^2) = \pi_b \mathcal{N}(x;\mu_b,\sigma_b^2).\]
Scores near this intersection are the most uncertain because the two components assign them similar likelihood. The system searches for unlabeled points in this uncertainty region and asks the user to classify them as anomalies or normal points. To keep the interaction manageable, the implementation limits each review round to at most 20 candidate points. If too many points are found, the uncertainty window is narrowed; if too few are found, it is widened.

The active-learning loop has two correction steps. First, if a point previously marked as an anomaly has a low score below the top-\(\rho\) cutoff, the system asks whether it should be removed from the anomaly set. This catches possible false anomaly labels. Second, the system asks the user to label points near the mixture intersection, because these points have the greatest effect on the threshold. After each feedback round, the mixture model is refit and the threshold is updated. The loop stops when the score distribution becomes stable or when no additional uncertain candidates are found.

\section{Pattern-Specific Thresholds for AnDri}

The AnDri workflow extends the thresholding procedure by estimating a separate threshold for each normal pattern represented in the training set. Let \(A\) be the set of user-selected training indices. For each normal pattern \(k\), the training indices assigned to that pattern are
\[A_k = \{i \in A \mid g(i)=k\}.\]
The system fits the same Bayesian mixture and active-learning procedure on the scores in \(A_k\), using only the user labels associated with that normal-pattern group. This produces a threshold \(\theta_k\) for normal pattern \(k\).

The final AnDri decision is therefore
\[
\hat{y}_i =
\begin{cases}
1, & s_i > \theta_{g(i)},\\
0, & s_i \leq \theta_{g(i)}.
\end{cases}
\]
This design is important under concept drift. If the time series contains several normal patterns, each normal pattern may have a different score range. A single global threshold may over-flag a drifting normal region or under-detect anomalies in a lower-variance region. Pattern-specific thresholds allow the decision boundary to follow the changing normal context learned by AnDri.

If a point belongs to a normal pattern that was not present in the user-selected training region, the implementation transfers the threshold from the most similar trained normal pattern. The nearest trained pattern is selected by Euclidean distance between normal-pattern subsequences:
\[k^*=\arg\min_{j \in K_{\text{train}}} \|NM_k-NM_j\|_2,\]
and the missing threshold is assigned as \(\theta_k=\theta_{k^*}\). This fallback allows the system to still make a decision for regions not directly covered by user feedback.

This methodology connects the algorithmic score output with the interactive system workflow. The backend produces normalized scores and, for AnDri, normal-pattern assignments. The frontend uses those values to guide user review, update thresholds, and present final anomaly decisions. In this way, the system does not rely only on a static cutoff. Instead, it uses partial human feedback to adapt the decision boundary and uses AnDri's normal-pattern structure to make the threshold sensitive to concept drift.

\section{System Architecture}

AnomDrift is a browser-based interactive system that uses Python Flask as the backend and natural JavaScript as the frontend. This system consists of three main layers: The first layer is the user interface in \textbf{templates/index.html}, which includes the visual layout, upload dialog window, control panel, algorithm tabs, and ECharts-based visualizations \cite{apacheecharts}. The second layer is the Flask API layer, implemented through \textbf{app.py} and the blueprint in \textbf{blueprints/detect\_anomalies/routes.py}. The third layer is the detection and scoring layer, implemented by \textbf{utils/ml.py} and the \textbf{algorithms/andri/} package. The scoring layer integrates AnDri with baseline methods, including SAND and DAMP from the TSB-UAD benchmark suite, and optionally NormA when the external confidential implementation is available \citep{boniol2021norma,boniol2021sand,lu2022damp,paparrizos2022tsbuad}.

\noindent The implemented architecture separates interaction, routing, and computation. The browser reads the selected or uploaded dataset, stores the time-series values, labels, user-selected training regions, current flags, and threshold state, and renders all charts. The backend is responsible for computational work such as running AnDri and baseline detectors, fitting the Bayesian Gaussian mixture model for user labels, computing AUC values, and finding similar normal patterns using time-series similarity search.

This architecture keeps the backend stateless and avoids the need for a database or session. Each detection request sends the current newest state to the server. The server then computes and returns the result to the browser. The architecture also reflects the goal of the project. The front-end  preserve states to show how the user selected training regions, how candidate labels were updated, how thresholds changed, and how each algorithm behaved on the same dataset.



\section{System Workflow}

Figure~\ref{fig:52} summarizes the complete workflow of the AnomDrift. The workflow begins with loading a dataset and setting the allowed anomaly proportion. The user then selects training regions from the time series and marks initial anomaly points through the interactive frontend. After these user inputs are collected, the backend computes anomaly scores by learning temporal normal patterns, updating normal-pattern states online, and assigning point-level anomaly scores. The frontend then supports an active-learning feedback loop, where the user reviews candidate points, corrects possible labeling mistakes, and adds missing anomaly labels. Finally, the system uses the refined threshold to detect anomalies and concept drifts, and presents both quantitative metrics and visual comparison results.

\subsection{Selecting Regions and Marking Anomaly Points}

The user selects training regions in the next stage. Figure~\ref{fig55} shows the region-selection interface. The implementation was designed to make this step easy to use even for long time series. The upper chart gives a global overview, and the lower chart gives a detailed view of the selected range. In the JavaScript implementation, the lower chart uses ECharts' inside data zoom, so the user can use the mouse wheel to zoom in and out directly on the chart. The selected training interval is implemented as a horizontal brush region. The user can drag the brush edges to resize the interval, and the frontend updates the stored \textbf{brushAreas} array immediately after the brush event. The user can also click a location in the global chart to navigate the detailed chart to that region. The user can select training regions visually instead of manually entering start and end indices.

After the user finishes selecting the training regions, the interface changes to the anomaly-marking phase, as shown in Figure~\ref{fig56}. The user can click the global chart to navigate to a local region, use the mouse wheel to inspect the detailed chart at a suitable scale, and then drag directly on the lower chart to mark sequence anomalies. In the JavaScript code, the anomaly-marking button enables a sequential marking mode. While this mode is active, the system temporarily disables chart zooming, listens for mouse down, mouse move, and mouse up events, draws a temporary rectangle over the selected range, converts the selected pixel range back to time-series indices, and updates the corresponding entries in the browser-side \textbf{flags} array as anomaly labels. The red points in the chart indicate the points that have been marked by the user. The interface also provides an undo button, which makes the labeling process more forgiving when the user selects an incorrect region. Here, users can use the right mouse button to unmark the anomalies. This is done to facilitate users' more precise control.

Because both training selection and anomaly marking are completed through mouse-based chart interactions, the system reduces the need for manual index lookup and makes the workflow more accessible to users who want to inspect the time series visually.

The way to find a similar normal pattern is as follows. For a candidate subsequence $q$ and a set of learned normal patterns $\mathcal{N}=\{N_1,N_2,\ldots,N_m\}$, the backend searches for the normal pattern and alignment position with the smallest subsequence distance:
\[
(N^{*}, \tau^{*}) =
\arg\min_{N_j \in \mathcal{N},\, \tau}
d\bigl(q, N_j[\tau:\tau+|q|-1]\bigr).
\]
Here, $d(a,b)$ denotes the z-normalized Euclidean distance between two equal-length subsequences $a$ and $b$:
\[d(a,b)=\|z(a)-z(b)\|_2,\]
where $z(\cdot)$ normalizes a subsequence to zero mean and unit variance. In the implementation, this distance profile is computed using \texttt{stumpy.mass}, which is provided by the STUMPY time-series data mining library \citep{law2019stumpy}. The selected normal pattern $N^{*}$ is then aligned at position $\tau^{*}$ and overlaid on the candidate region.




The middle chart shows the positions of the candidate points in a broader time series. This view answers another question: Where exactly do these candidate points occur within the entire dataset? By displaying the candidate points along with their surrounding values, users can determine whether the suspicious points occur independently, repeatedly, or in the drift area of the time series.

After every user decision, the frontend updates the browser-side label state and calls the label-fitting function again. As a result, both the TP/FP pie chart and the anomaly-score distribution with the threshold line are refreshed immediately. This gives the user direct visual feedback about the effect of the latest decision: the pie chart shows whether the decision reduces false positives or confirms more true positives, while the updated threshold display shows how the decision changes the current separation between normal and anomalies scores.

This iterative update is useful because the user does not need to wait until the end of the workflow to understand whether the feedback is helping. Each click on the \textit{Anomalies} or \textit{Normal Points} button changes the label set, recomputes the score-based threshold, and updates the visual summaries. Through this loop, the threshold is refined from user feedback rather than being treated as a fixed value chosen before visualization.

\section{Backend Architecture}
The backend is implemented in Flask. The root route in \textbf{app.py} creates the main page and checks whether the optional NormA implementation exists under \textbf{algorithms/norma/norma.py}. This check is necessary because the NormA source file is not included directly in the repository. To use the source file, the user must get it from the author. AnDri, SAND, and DAMP are available.

The backend receives JSON format data for each analysis request. For the detector endpoints, the JSON content includes the time-series values, current flags, training-set ranges, hyperparameters, and baseline labels. The backend converts these values to normalized scores in the range \([0,1]\), and returns lists that can be rendered directly by the front-end.

The scoring functions in \textbf{utils/ml.py} are designed as wrappers around the underlying detector implementations. For AnDri, the wrapper constructs an \textbf{AnDri} object, calls its \textbf{fit} method, extracts the score sequence, extracts normal-pattern subsequences, and returns the normal-pattern index associated with each point. For SAND and DAMP, the wrappers call the corresponding TSB-UAD models and normalize their decision scores. For NormA, the wrapper imports the optional external implementation and raises a clear error if it is unavailable.

The backend also supports the active-learning loop. The \textbf{fit\_user\_labels} function fits a two-component Bayesian Gaussian mixture model to the currently marked anomaly scores. It returns whether the score distribution is stable. If the score distribution is not stable, it also returns the uncertain candidates. The front-end then uses this information to ask the user for more labels. 


\section{Deployment}
\label{sec:D}

\textcolor{blue}{The system was deployed on a virtual machine hosted on our lab server running Red Hat Enterprise Linux 9.8 to provide convenient access. The deployment environment was allocated four virtual CPU (vCPU) cores and 62GB of memory, with the underlying hardware based on an Intel Xeon E5-4640 processor operating at 2.40GHz. The supervisor and lab-mates can view the demo through their own browsers. This deployment also provides practical opportunities for the study of software engineering. Building the local demo requires integrating algorithms, data processing, and visualization functions; while the deployment process requires understanding a series of different engineering concepts, including Web server configuration, reverse proxy, static resource processing, dependency management, request size limit, file permissions, server path, process restart and runtime logs.}

The entry point of the web application is the Flask application defined in \textbf{app.py}. When starting, the application creates a Flask instance, registers the backend blueprints, sets the maximum upload size for time-series files, and provides the main interface through the root route. The system also checks whether the optional NormA implementation exists under \textbf{algorithms/norma/norma.py}. This is necessary because the NormA implementation is not directly included in the code repository, while the remaining algorithms can still be used without it. This design enables the same interface to support different baseline algorithms based on the available components in the runtime environment.

Figure~\ref{fig591} illustrates the server-side deployment pattern. In this architecture, browser requests first reach NGINX, which acts as the reverse proxy. NGINX is typically used as an HTTP server and reverse proxy, and it can also efficiently provide static resources \citep{nginx}. Dynamic requests are then forwarded to the Python application process managed by Gunicorn, which is a Python WSGI HTTP server that can run Flask applications under a production environment \citep{gunicorn,flask}.

    \centering

This separation is useful because static files can be handled and cached independently, while computational requests such as dataset loading, anomaly detection, label submission, threshold recalculation, and similar-pattern search are processed by the backend.

This chapter describes the implementation of the AnDri demonstration system. The system was designed to let a user upload or select a time-series dataset, inspect the data visually, select training regions, mark anomalies, run AnDri and baseline detectors, review uncertain candidates, and compare the final detection results. Interactive visualizations are integrated into each stage of the workflow.


\section{System Overview and Architecture}

AnDri is a browser-based interactive system with a Python Flask backend and a nature JavaScript front-end. The system has three main layers. The first layer is the user interface in \textbf{templates/index.html}, which contains the visual layout, upload dialog window, control panel, algorithm tabs, and ECharts-based visualizations \cite{apacheecharts}. The second layer is the Flask API layer, implemented through \textbf{app.py} and the blueprint in \textbf{blueprints/detect\_anomalies/routes.py}. The third layer is the detection and scoring layer, which is implemented in \textbf{utils/ml.py} and the \textbf{algorithms/andri/} package. The scoring layer integrates AnDri with baseline methods, including SAND and DAMP from the TSB-UAD benchmark suite, and optionally NormA when the external proprietary implementation is available \citep{boniol2021norma,boniol2021sand,lu2022damp,paparrizos2022tsbuad}.

\noindent The implemented architecture separates interaction, routing, and computation. The browser reads the selected or uploaded dataset, stores the time-series values, labels, user-selected training regions, current flags, and threshold state, and renders all charts. The backend is responsible for computational work such as running AnDri and baseline detectors, fitting the Bayesian Gaussian mixture model for user labels, computing AUC values, and finding similar normal patterns using time-series similarity search.

\begin{figure}[H]
    \centering

    \caption{Overall workflow of the AnDri demo system \citep{park2025andri}.}
    \label{fig52}

This architecture keeps the backend stateless and avoids the need for a database or session. Each detection request sends the current newest state to the server. The server then computes and returns the result to the browser. The architecture also reflects the goal of the project. AnDri is an interactive visual analytics system for co-exploring anomalies and concept drift. Therefore, the front-end must preserve enough state to show how the user selected training regions, how candidate labels were updated, how thresholds changed, and how each algorithm behaved on the same dataset.


\section{Front-End Interface and Visualization}

The front-end is organized around a left control panel and a right visualization area. The control panel guides the user through the workflow. The implementation uses Apache ECharts for browser-side rendering \citep{apacheecharts}. ECharts can render line charts, pie charts, and histograms in the frontend browser, and supports zoom controls, brush area selections, highlighted regions, and marking points within selected areas. In the implementation, the full-data chart uses progressive rendering thresholds for long series, and both charts use interactive data zoom so that users can move between the full context and local details. Training regions are shown as shaded areas, candidate anomalies are colored, and score thresholds are drawn directly on score charts. These design choices make the workflow visual rather than script-driven.

After selecting training regions, the user starts to label the anomalies. The interface allows the user to drag over the detailed chart to mark sequential anomaly regions. The selected anomalies are stored as labels in the browser-side flag array. The system can then use these labels to estimate an initial decision threshold and to identify candidate points that require further review.

Then the user will be asked to classify uncertain candidates as anomalies or normal points. The histogram shows the distribution of marked anomaly scores and the current threshold. The time-series panels show candidate points in context, and normal-pattern overlays help the user understand whether a suspicious region differs from expected behavior.

\section{Backend Architecture}
The backend is implemented in Flask. The root route in \textbf{app.py} creates the main page and checks whether the optional NormA implementation exists under \textbf{algorithms/norma/norma.py}. This check is necessary because the NormA source file is not included directly in the repository. To use the source file, the user must get it from the author. AnDri, SAND, and DAMP are available.

The backend receives JSON format data for each analysis request. For the detector endpoints, the JSON content includes the time-series values, current flags, training-set ranges, hyperparameters, and baseline labels. The backend converts these values to normalized scores in the range \([0,1]\), and returns lists that can be rendered directly by the front-end.

The scoring functions in \textbf{utils/ml.py} are designed as wrappers around the underlying detector implementations. For AnDri, the wrapper constructs an \textbf{AnDri} object, calls its \textbf{fit} method, extracts the score sequence, extracts normal-pattern subsequences, and returns the normal-pattern index associated with each point. For SAND and DAMP, the wrappers call the corresponding TSB-UAD models and normalize their decision scores. For NormA, the wrapper imports the optional external implementation and raises a clear error if it is unavailable.

The backend also supports the active-learning loop. The \textbf{fit\_user\_labels} function fits a two-component Bayesian Gaussian mixture model to the currently marked anomaly scores. It returns whether the score distribution is stable. If the score distribution is not stable, it also returns the uncertain candidates. The front-end then uses this information to ask the user for more labels. 


\section{Data Upload and Preprocessing Pipeline}
AnDri system supports both built-in example datasets and local user-provided datasets. The built-in examples are stored in \textbf{static/datasets/} and include Repeat ECG, Weather, and Electricity data. The available local formats are \textbf{.arff} and \textbf{.csv}. In the front-end parser, the system reads each row, extracts the time-series value and label from the last two columns, and stores them in two arrays: \textbf{fullData} for values and \textbf{baseline} for labels. After parsing, the system initializes the interactive charts and enters the training-set selection phase.

\section{Deployment}
The system was deployed on a school server to provide accessibility. The supervisors and lab-mates can inspect the demo through their own browser.  The deployment also served as a software engineering learning experience. Building the local demo required integrating algorithms, data processing, and visualization; deploying it required understanding a different set of engineering concepts, including web-server configuration, reverse proxy, static-resource handling, dependency management, request-size limits, file permissions, server paths, process restarts and runtime logs.

The application entry point is the Flask application defined in \textbf{app.py}. At startup, the application creates the Flask instance, registers the backend blueprints, sets a maximum upload size for large time-series files, and serves the main interface through the root route. The latest repository also checks whether the optional NormA implementation exists under \textbf{algorithms/norma/norma.py}. This is necessary because the NormA implementation is not included directly in the repository, while the remaining algorithms can still be used without it. This design allows the same interface to support different baseline algorithms depending on which optional components are available in the runtime environment.

Figure~\ref{fig58} illustrates the server-side deployment pattern. In this architecture, browser requests first reach NGINX, which acts as the reverse proxy. NGINX is commonly used as an HTTP server and reverse proxy, and it can also serve static resources efficiently \citep{nginx}. Dynamic requests are forwarded to the Python application process managed by Gunicorn, a Python WSGI HTTP server that can run Flask applications in a production-style environment \citep{gunicorn,flask}.

This separation is useful because static files can be handled and cached independently, while computational requests such as dataset loading, anomaly detection, label submission, threshold recalculation, and similar-pattern search are processed by the backend.
